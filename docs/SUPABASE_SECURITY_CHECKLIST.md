# MYRA — чек-лист безопасности Supabase

Технический разбор в дополнение к issue **#43 «Эпик: Безопасность»** — здесь
конкретика по факту чтения `supabase/schema.sql` и `supabase/functions/*` на дату
написания, не абстрактные пункты.

## RLS — построчная сверка (`supabase/schema.sql`)

Все 14 секций схемы имеют `alter table ... enable row level security` и хотя бы
одну политику на каждую операцию, которая вообще разрешена клиенту. Проверено построчно:

| Таблица | RLS | Публичный SELECT | Заметка |
|---|---|---|---|
| `profiles` | ✅ | да (`using (true)`) | email вынесен в `profile_private` намеренно — не читаем публично |
| `profile_private` | ✅ | нет, только владелец | |
| `tracks` | ✅ | да, кроме `hidden=true` (кроме владельца/админа) | |
| `comments` | ✅ | да | insert-only, нет update/delete — сознательно (append-only) |
| `donations` | ✅ | нет, только `from_user`/`to_user_id` | |
| `subscriptions` | ✅ | нет, только владелец | `update` явно запрещает клиенту выставить `status = 'active'` себе — только сервисный ключ |
| `follows` | ✅ | да | |
| `admins` | ✅ | только своя строка | **нет insert/update/delete-политик вообще** — заполняется вручную через Table Editor, это осознанный барьер против самоназначения в админы |
| `support_messages` | ✅ | владелец треда + админ (через `exists` на `admins`) | |
| `user_follows` | ✅ | только свои подписки (`follower_id`) | публичного счётчика подписчиков сейчас нет |
| `reports` | ✅ | владелец жалобы + админ | |
| `payments` | ✅ | только владелец, **insert/update отсутствуют для authenticated/anon** | пишет только service-role из edge-функций |
| Storage `tracks` bucket | ✅ (policies on `storage.objects`) | read публичный | write ограничен `{auth.uid()}/...` через `storage.foldername(name)` |

**Не найдено:** таблиц без RLS, политик с `using (true)` там, где это не задумано
явно, публичного read/write для `admins`/`payments`.

## GRANT — сверка с политиками

Каждая секция схемы сопровождается блоком `grant ...` (см. схему целиком — GRANT
контролирует доступ к таблице **до** RLS, отдельно от него). Все GRANT на дату
написания соответствуют политикам один в один (например, `payments` — только
`select` для `authenticated`, `insert`/`update` никому, ровно как задумано).
Если при будущих правках добавляется новая таблица — не забыть парный `grant`,
иначе будет `permission denied for table`, даже с правильными RLS-политиками.

## Edge Functions — двухклиентский паттерн

| Функция | Аутентификация вызывающего (anon+JWT) | Привилегированная запись (service-role) | Особенность |
|---|---|---|---|
| `create-payment` | ✅ `auth.getUser()` | ✅ | |
| `delete-account` | ✅ | ✅ | |
| `set-subscription` | ✅ | ✅ | именно эта функция — единственный легитимный способ выставить `subscriptions.status = 'active'`, т.к. RLS клиенту это прямо запрещает |
| `support-chat` | ✅ | не используется — пишет как сам пользователь через RLS | обращение к внешнему API (`OPENROUTER_API_KEY`), есть rate-limit |
| `yookassa-webhook` | не применимо (публичный, без JWT — так и должно быть) | ✅ | **не доверяет `status` из тела webhook** — независимо перезапрашивает статус `GET /v3/payments/{id}` своим секретным ключом перед записью; идемпотентен к повторной доставке |

Все проверено: ни одна функция не создаёт service-role клиент раньше, чем
подтверждён JWT вызывающего (кроме `yookassa-webhook`, где это архитектурно не нужно —
см. пояснение в самом файле функции).

      перед публичным релизом (`git log -p -- '*.env*'` и секрет-сканер GitHub, см. issue #43)
- [ ] Rate-limit сейчас есть только на `support-chat` (единственная функция, вызывающая
      платный внешний API) — `yookassa-webhook` защищён иначе: не доверяет телу
      запроса и всегда перепроверяет статус у ЮKassa, а не частотой вызовов
