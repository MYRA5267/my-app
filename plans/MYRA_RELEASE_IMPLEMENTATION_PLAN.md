# MYRA — план доведения проекта до публичной Beta и Release 1.0

## 1. Назначение документа

Этот файл является главным источником истины для дальнейшей реализации MYRA.

Оркестратор обязан:

— читать этот файл целиком при каждом новом запуске;
— определять состояние по чекбоксам, git log, PR, Issues и фактическому коду;
— не повторять завершённую работу;
— обновлять статусы после каждого законченного этапа;
— фиксировать принятые решения;
— фиксировать ограничения и действия владельца;
— оставлять точную точку продолжения при завершении контекста.

Цель — довести MYRA до состояния:

1. Стабильная Web Beta.
2. Устанавливаемая и проверенная PWA.
3. Android-приложение с настоящим фоновым воспроизведением.
4. Воспроизводимая Android-сборка для внутреннего тестирования.
5. Подготовка подписанных APK и AAB.
6. Подготовка к RuStore и Google Play.
7. Подготовленная iOS-архитектура и TestFlight-процесс.
8. Безопасный Supabase backend.
9. Юридические и служебные страницы.
10. Проверенный процесс работы с музыкальными правами.
11. Автоматические тесты и CI.
12. Честное разделение реальных и демонстрационных функций.

## 2. Статусы

Использовать следующие статусы:

— NOT_STARTED — работа не начиналась;
— IN_PROGRESS — работа выполняется;
— REVIEW — реализация закончена, идут проверки;
— BLOCKED_OWNER — требуется действие владельца;
— BLOCKED_EXTERNAL — внешний технический блокер;
— DONE — все критерии готовности выполнены.

Для каждой фазы обязательно указывать:

— текущий статус;
— дату последнего изменения;
— SHA последнего относящегося коммита;
— краткое фактическое состояние;
— следующий точный шаг.
