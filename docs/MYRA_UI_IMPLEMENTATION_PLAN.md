# MYRA — план внедрения DETAIL и премиум-редизайна

> Приложенное изображение брендборда используется **только как визуальный референс**.
> Главный объект для внедрения — фрагмент **DETAIL** (правая панель брендборда:
> светящаяся полупрозрачная волна в медно-розовых/золотистых/фиолетовых/холодных
> тонах), а не брендборд целиком и не логотип-леттеринг в его текущем виде.

Этот документ — аудит текущего состояния и план работ по превращению MYRA в
цельный премиальный продукт с фирменным визуальным мотивом DETAIL. Он **не**
описывает уже выполненную работу — только то, что предстоит сделать, поэтапно,
с проверяемыми контрольными точками.

Статус на момент написания: **baseline подтверждён зелёным**
(`pnpm typecheck` и `pnpm build` прошли чисто до каких-либо изменений —
результаты в разделе «Baseline» ниже). Масштабные правки не вносились.

### 1.2 Уже существующий визуальный язык (не с нуля)

MYRA уже не использует «дефолтные» компоненты — есть узнаваемый слой:

- **`THEMES`** (`lib.tsx`) — три темы (`dark`/`light`/`neon`, неон закрыт
  апгрейдом), каждая со своим `--brand-grad` (фиолет → небо → жемчужная роза),
  `--cover-filter`, `--glass-*`, `--aurora-fade`. Применяются как inline-style
  на корне, без `.dark`-класса.
- **`Aurora`** (`lib.tsx:482`) — уже существующий «живой фон»: три
  radial-gradient пятна с drift-анимацией (`drift1/2/3`, определены в
  `App.tsx` в глобальном `<style>`, не в CSS-файлах). Используется в
  `FullPlayer`, `DynamicBg`, hero Прилива на главной, онбординге — это и есть
  ближайший функциональный аналог того, что должен делать DETAIL.
- **`FrequencyOrb`** (`lib.tsx:895`) — объёмная стеклянная сфера-визуализатор
  с параллаксом, уже в `FullPlayer`.
- **`ParticleWave`** (`lib.tsx:592`) — canvas-скраббер частицами вместо
  DOM-баров, уже используется как основная волна прогресса.
- **`GLASS`** — единый стеклянный стиль (blur+saturate+edge-highlight).
- **`isWeakEnvironment()`** (`lib.tsx:130`) + класс **`fx-simple`** — уже
  работающий механизм деградации: гасит `backdrop-filter`, `.fx-heavy`
  (тяжёлые blur-обложки), `.fx-aurora` анимации, `.fx-parallax`, `.fx-orb`
  декор. Правила — в инлайновом `<style>{GLOBAL_STYLE}</style>` в `App.tsx`
  (не в `theme.css`!) — это нужно знать, чтобы не искать их не в том месте.
- **Reduced motion**: `isWeakEnvironment()` уже включает
  `prefers-reduced-motion` как один из триггеров упрощённой графики — то есть
  инфраструктура для п.9 ТЗ (анимации/reduced motion) уже частично есть,
  DETAIL нужно просто подключить к тем же двум переключателям
  (`fx-simple` класс + `document.querySelector(".fx-simple")` в rAF-циклах).

**Вывод**: DETAIL — это не новая система с нуля, а **замена/усиление Aurora**
в ключевых точках (Full Player, hero-блок главной), с сохранением всех
существующих gating-механизмов производительности.

## 2. Фирменный ассет DETAIL — статус и ограничение

**Технический факт**: у меня нет инструмента для растровой обработки
изображений (кроп/экспорт PNG/WebP из присланного референса) — только
чтение файлов и работа с кодом. Приложенное изображение доступно мне только
как визуальная референция в диалоге, не как файл на диске, который можно
программно обрезать.

Это значит по букве раздела 2 ТЗ («если автоматическое выделение
невозможно — создать временный ассет и указать, какой файл нужно заменить
вручную»):

**Что сделаю на Этапе 1**: процедурный CSS/SVG placeholder «в духе» DETAIL —
радиальные градиенты + одна-две органические волновые path-формы в
медно-розовых/золотых/фиолетовых/холодных тонах (палитра из уже существующих
`--brand-grad`/трек-акцентов, не нафантазированная заново), анимированный
теми же `drift`-принципами, что уже использует `Aurora`. Это даст рабочий,
проверяемый компонент `DetailBackdrop` сразу, без блокировки на файл.

**Что нужно от тебя (или дизайнера) отдельно**: реальный кроп DETAIL-фрагмента
из референса в виде файлов:

```
public/detail/detail-full.webp       (~1600×1200, для hero/Full Player)
public/detail/detail-full.png        (фолбэк)
public/detail/detail-soft.webp       (уменьшенный контраст/яркость, для тонких мест типа Library/Profile)
public/detail/detail-mobile.webp     (обрезанный по центру, компактный, ~800×800)
public/detail/detail-low.jpg         (маленький, сильно сжатый — для fx-simple/слабых устройств)
```

Когда файлы появятся — `DetailBackdrop` (Этап 1) переключится с
процедурного fallback на реальный ассет одной строкой (путь уже
параметризован), без переверстки компонента.

## 3. Риски

1. **Расхождение с `CLAUDE.md`**: ТЗ (раздел 13) предлагает переструктурировать
   `src/app/` в `src/features/*`. Это прямо противоречит задокументированному
   в `CLAUDE.md` архитектурному решению («flat files, not folders» —
   намеренный выбор, не забытая рефакторинг-задача) и является ровно тем
   «огромным рискованным рефакторингом одним коммитом», который сам же ТЗ
   (разделы 13 и 23) запрещает. **Рекомендация**: не делать структурную
   миграцию папок в этом цикле. Вместо этого — точечно выносить хуки
   (`usePlayerState`, `useAuthState` и т.п.) из `App.tsx` туда же, в `src/app/`,
   без смены каталогов, если и когда до этого дойдёт очередь (Этап 4). Прошу
   подтверждения перед началом этой части, т.к. это расходится с буквой ТЗ.
2. **61 `useState` в `AppInner`**: реальная проблема, но вытаскивание требует
   очень аккуратного порядка хуков (в кодовой базе уже был инцидент — краш
   «Rendered more hooks than during the previous render» из-за хука после
   раннего `return`, см. комментарий в `App.tsx:1124`). Рефакторинг стейта —
   отдельный этап (4), не смешивается с визуальными правками DETAIL.
3. **Производительность на Android WebView**: любой новый визуальный слой
   (DETAIL) обязан пройти через `fx-simple`/`fx-heavy` гейты с первого
   коммита, не «добавим потом» — иначе повторится ровно та проблема с
   мигающими элементами, которую уже чинили (см. `isWeakEnvironment()`
   комментарий).
4. **Бюджет бандла**: DETAIL-ассеты (WebP/AVIF) должны грузиться `lazy`/через
   `<img loading="lazy">` и не должны попадать в основной JS-чанк — иначе
   ухудшат уже отмеченное предупреждение Vite про >500kB чанки.
5. **Масштаб ТЗ (24 раздела)**: полный объём этого технического задания
   (PWA + Capacitor prep + Supabase security checklist + юридические страницы
   + аналитика + тестирование + структурный рефакторинг + весь визуальный
   редизайн) — это работа на много отдельных PR, не один присест. План ниже
   разбивает её так, чтобы после каждого этапа продукт оставался рабочим и
   собираемым, но полный охват всех 24 разделов ТЗ потребует нескольких
   последовательных циклов такого масштаба, каждый со своим планом/аудитом.
