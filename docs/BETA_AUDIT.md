# MYRA — аудит: реальные данные vs демо-каталог

Снимок на дату написания. Цель — явно перечислить, что в приложении **по-настоящему
пользовательское** (живёт в Supabase, переживает перезагрузку, видно другим людям),
а что **демо/статика** (нужно только для того, чтобы приложение работало красиво и
осмысленно без единой настройки бэкенда — см. CLAUDE.md, «MYRA... должно всегда
работать как полностью локальное/офлайн демо»). Ни один пункт ниже не наблюдение
«как должно быть» — это то, что реально в коде на сейчас.

## Правило по умолчанию

Если `supabaseEnabled === false` (нет `VITE_SUPABASE_URL`/`VITE_SUPABASE_ANON_KEY` —
дефолт в dev/CI и при первом локальном запуске без настройки) — из перечисленного
ниже работает **только «Демо»**-колонка. «Реальное» либо не отображается совсем,
либо использует локальный резерв (`localStorage`, эмуляция), явно помеченный как
таковой (см. «Принцип честности» в CLAUDE.md — `don.simNote`/`plus.simNote` и т.п.).

## Каталог и контент

| Данные | Источник | Реально или демо |
|---|---|---|
| `TRACKS`, `ARTISTS`, `CHARTS`, `PLAYLISTS` (8 демо-треков и т.д.) | `src/app/data.ts`, статичный массив | **Демо.** Никогда не пишется в базу, id — маленькие числа |
| Треки, опубликованные через форму релиза (Студия) | `public.tracks` (Supabase) | **Реальные.** `owner_id` = аккаунт автора, RLS на insert/update/delete |
| Обложки/аудиофайлы загруженных треков | Supabase Storage, бакет `tracks` | **Реальные** (публичный read, запись — только в свою папку `{uid}/...`) |
| Комментарии на волне трека | `public.comments`, `track_id` — `text` | **Смешанный ключ**: либо реальный `tracks.id`, либо `"catalog:N"` для демо-трека — намеренно (см. CLAUDE.md, «Граница модели данных»), не баг |
| Жалобы (модерация) | `public.reports`, `target_id` — `text` | Та же смешанная природа ключа, что и у комментариев |

## Аккаунты и профиль

| Данные | Источник | Реально или демо |
|---|---|---|
| Имя/роль при офлайн-онбординге | `localStorage` (`ls` в data.ts) | **Демо-режим**: живёт только на этом устройстве, не хендл, не «настоящий» аккаунт |
| Аккаунт после signup/login с Supabase | `auth.users` + `public.profiles` + `public.profile_private` | **Реальные.** Email вынесен в приватную таблицу отдельно от публичного `profiles` — см. схему, секция 1b |
| Достижения (Профиль → «Достижения») | `achievements.ts`, считается из локальной статистики прослушиваний | **Демо/локальные** — не персистятся в Supabase, теряются при смене устройства |
| Статистика прослушиваний (XP, Эхо месяца, сплит) | `stats.ts`, `localStorage` | **Локальные**, не синхронизируются между устройствами одного аккаунта |

## Социальный слой

| Данные | Источник | Реально или демо |
|---|---|---|
| Подписка на демо-артиста каталога (`follows`) | `public.follows`, `artist_name` — `text` | Реальная строка в базе, но указывает на **демо-имя**, не на аккаунт |
| Подписка человека на человека (`user_follows`) | `public.user_follows`, `followee_id` — `uuid` | **Полностью реальные** аккаунты с обеих сторон |
| Совместные комнаты (rooms.ts) | Supabase Realtime Broadcast/Presence, без таблицы БД | **Реальные** на время сессии — код комнаты и есть канал, история не сохраняется |
| Чат поддержки | `public.support_messages` | **Реальный**, если Supabase настроен; иначе — `sup.offlineNote`, сообщение остаётся только в этом браузере |

## Деньги

| Данные | Источник | Реально или демо |
|---|---|---|
| Донат/подписка через ЮKassa (настроена) | `public.payments`, Edge Functions `create-payment`/`yookassa-webhook` | **Реальные деньги.** Статус пишет только сервисный ключ, после независимой проверки у ЮKassa |
| Донат/подписка без настроенной ЮKassa | Симулированная карта/QR-флоу | **Явно помечено** как симуляция (`don.simNote`/`plus.simNote`/`wd.simNote`) — деньги никуда не уходят |
| `public.donations` (демо-артисту, `to_artist` text) | Таблица БД | Строка **реальная**, но получатель — **демо-имя**, реальному человеку ничего не зачисляется |
| `public.donations.to_user_id` (после секции 10 схемы) | Таблица БД | **Реальный получатель**, если донат шёл настоящему артисту с аккаунтом |

## Что проверить перед публичным бета-релизом

- [ ] Нигде в UI демо-данные не подписаны так, будто они реальные (статистика
      прослушиваний, достижения — они уже читаются с локального устройства, а не
      с сервера; убедиться, что в текстах нет формулировок вида «твой глобальный рейтинг»)
- [ ] Явный визуальный маркер для симулированных денежных операций сохранён во всех
      точках входа (сейчас: донат, MYRA Plus/Pro, вывод — `wd.simNote`)
- [ ] `catalog:N`-ключи в `comments`/`reports` не путают модерацию: жалоба на
      демо-контент не должна попадать в очередь так, будто это чей-то реальный трек
- [ ] Если демо-каталог (`data.ts`) когда-либо получит собственные аккаунты-артисты —
      явно решить миграцию `follows`/`donations` с text-имени на `uuid`, не молча
